# 003: バグ管理番号が未記載のテスト不具合(7件)の紐付け確認

- 種別: ドキュメント整合性確認
- 優先度: 低
- ステータス: 完了

## 背景・現状

- システムテスト設計書(1回目実施、NG110件)のうち、03_認証・認可シートのid 78, 83, 84, 89, 92, 93, 100 の7件は、NG内容は記録されているものの対応するバグ管理表のバグ番号が空欄のまま残っている(管理者の自己削除確認・エラー画面遷移確認関連の項目)。
- 他のNG項目は全てバグ管理表の28件のいずれかに紐づいており、この7件だけが追跡上の対応関係が不明瞭になっている。

## 対応関係

システムテスト設計書の該当7件を確認したところ、いずれも「クエスト/支出上限の承認・却下・完了申請・削除」系エンドポイントに対する認可テストで、大きく2パターンに分かれることが分かった。

- **パターンA(他人・他家族の"有効な"IDを指定)**: バグ管理表の該当バグ番号が明確に存在し、バグ番号欄が空欄なのは単なる記載漏れ。
- **パターンB(存在しない"999"のようなID を指定)**: パターンAとは別の観点のテストで、バグ管理表にはこのパターン専用のバグ票が存在しない。パターンAのバグ修正時に同じ関数を直しても、実装方法によっては直っていないことがあり、実際に**id=92, 93の2件は未修正のまま**だった。

| test id | 内容 | 対応するバグ | 種別 | 修正状況 |
|---|---|---|---|---|
| 78 | 自分に紐づかない子供の支出上限申請を却下 → エラー画面遷移確認(action: `/parent/limit/8/reject`) | **No.28**(支出上限申請却下(保護者)) | パターンA | チケット002で修正済み |
| 83 | 自分の子供でないクエストの完了申請 → エラー画面遷移確認(action: `/child/quest/6/complete`) | **No.10**(クエスト機能の権限制御(子供)) | パターンA | チケット002で修正済み |
| 84 | 管理者が自分自身のアカウントを削除 → エラー画面遷移確認(action: `/admin/accounts/9/delete`) | **No.24**(管理者が自身のアカウント削除可能) | パターンA | チケット002で修正済み |
| 89 | 存在しない(999)クエストを削除 → エラー画面遷移確認(action: `parent/quest/999/delete`) | **No.8**(クエスト管理機能の権限制御(保護者))と同一エンドポイント | パターンB | チケット002の`deleteQuest`修正(`orElseThrow`化)の副次効果で結果的に解消済み |

補足: バグ管理表の「関連テストケースID」欄(例: No.8は"認証認可(83,84,"のように記載)は、実際のテスト内容と付き合わせると対応が取れていない/末尾が欠落している箇所があり、この欄自体の記載精度も低いことが分かった(本チケットでは欄の訂正までは行っていない)。

## 実施内容(id=92, 93 の追加修正)

`QuestService.approveQuest` / `rejectQuest` は、チケット002で他家族の子供のクエストに対する所有者チェックを追加したが、**対象クエストがそもそも存在しない場合は `Optional#ifPresent` で静かに何もせず正常リダイレクトしてしまう**実装のままだった(存在確認自体は行われるが、「見つからない」ケースで例外を投げず、コントローラは承認/却下が成功したかのように `PARENT_APPROVALS` へリダイレクトしていた)。これは期待動作(エラー画面遷移)と異なるため、他のメソッド(`updateQuest`, `deleteQuest`, `requestComplete`)と同じ `findById(...).orElseThrow(() -> new IllegalArgumentException(...))` パターンに統一した。

### 検証方法

アプリを起動し、保護者アカウントでログインした状態で、存在しないクエストID(999)に対して `/parent/approvals/999/approve` と `/parent/approvals/999/reject` をそれぞれPOSTし、レスポンスが正しく「エラー画面」(`user-session-error.html`)としてレンダリングされることを確認した(修正前は通常のリダイレクトで正常終了していた)。
